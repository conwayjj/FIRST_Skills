Image Recognition

A simulated camera hands you a low-resolution grid of pixels — some carrying a hue/saturation/value color, some carrying a fiducial (AprilTag-style) ID. Write a Java void scan() { ... } that walks the frame pixel-by-pixel and calls report(...) with the targeting data your robot would act on — exactly the kind of loop a real vision co-processor runs every frame.

Objective

Turn pixels into targeting data

In Color Blob mode, every pixel carries a hue (0–360°) and a saturation (0–1); the target object is the region whose hue sits near targetHue with high saturation, while background and lighting only vary in brightness (getVal) — a correct detector should ignore brightness entirely. Hue also wraps at 360°, so red (near 0°) needs a wrap-aware distance check. A stray noisy pixel or two shouldn't be enough to report a detection — that's what the Robust Blob example's minimum pixel count guards against. In AprilTag mode, tags hand you a stable integer ID directly through getTagId(x, y) — no color math needed, just averaging the pixels that share an ID. Switch modes, add clutter, and see how your scan() holds up.

Reference

scan() API

width, height
Frame dimensions in pixels — loop 0..width-1 and 0..height-1.
getHue(x, y)
Hue at pixel (x, y), 0–360°. Meaningless on low-saturation (gray) pixels.
getSat(x, y)
Saturation at pixel (x, y), 0 (gray) to 1 (vivid color).
getVal(x, y)
Brightness at pixel (x, y), 0–1 — varies with lighting/vignette, ignore for color matching.
getTagId(x, y)
Fiducial ID at pixel (x, y), or -1 if this pixel isn't part of a tag.
targetHue
The hue (0–360°) of the color currently selected in "Track."
report(cx, cy, count)
Report a color-blob detection: centroid pixel coordinates and matched pixel count.
reportTag(id, cx, cy)
Report a tag detection: its ID and centroid pixel coordinates.
println(...)
Write a line to the console panel below the editor.
